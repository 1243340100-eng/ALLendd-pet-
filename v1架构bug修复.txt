一、已经确认的实际问题

实际 SQLite 中：

- users/settings 已正确保存用户填写的 nickname 和 preferred_name。
- 但 memories 中出现了错误记录：
  - “用户昵称为昌昌”
  - “用户的名字是昌昌”
- character-packs/default/persona.json 的 corePrompt、speakingStyle、sampleDialogues、memoryGuidance 和 userPetName 多处硬编码“昌昌”。
- GraphDispatcher.handleChatDirect() 每轮直接使用 pack.persona，没有加载 users/settings 中的用户称呼。
- OnboardingGraph 构建出的自定义 Persona 只存在于当次 Graph state，没有成为后续 ConversationGraph 的运行时 Persona。
- ReflectionGraph 同时把 userMessage 和 assistantReply 发给模型，未要求候选记忆必须有来自用户原文的证据，所以 AI 自己说出的错误内容会被当作用户事实写入。

- route-or-extract.ts 明确不调用模型，提醒提取完全依赖正则。
- ConversationGraph 的系统 Prompt 没有当前时间。
- SQLite memories.created_at/updated_at 使用 datetime('now')，虽然精确到秒，但格式没有 Z 或时区，UI 也没有明确显示本地写入时间。

五、P0：智能提醒解析

采用混合架构：

user input
→ 本地高召回候选检测
→ fastModel 结构化解析
→ TimeService 验证和归一化
→ 缺字段则 checkpoint 追问
→ PermissionGuard
→ create_reminder skill
→ SQLite transaction
→ 成功回复

本地规则只负责判断“可能是提醒请求”，不要继续独自承担完整时间解析。

候选包括：

- 提醒我……
- 记得叫我……
- 到时候告诉我……
- 一会儿提醒一下……
- 别让我忘了……
- N 分钟/小时后……
- 明早/今晚/下周……
- 用户正在补充 reminder checkpoint 时的短输入。

调用 ModelGateway 的 fastModel，输出严格 JSON：

{
  "intent": "create_reminder|not_reminder",
  "content": "标准化提醒内容",
  "originalTimeExpression": "一分钟后",
  "triggerAtUtc": "ISO-8601 UTC",
  "timezone": "Asia/Shanghai",
  "isRepeating": false,
  "recurrenceRule": null,
  "priority": "low|normal|high",
  "missingFields": [],
  "confidence": 0.0,
  "assumptions": []
}

模型 Prompt 必须包含可信时间：

- 当前 UTC 时间
- 当前本地时间，精确到秒
- IANA 时区
- 星期
- 当前本地日期
- 用户原始输入
- 已有 reminderDraft
- 本轮消息接收时间

示例：

当前本地时间：2026-07-11 16:07:12
时区：Asia/Shanghai
用户：你可以在一分钟后提醒我喝水吗

期望：

content = "喝水"
triggerAtUtc 对应本地 16:08:12
missingFields = []
confidence 足够高时直接创建，不追问。

模型不得直接写数据库。TimeService 必须验证：

1. ISO 格式合法。
2. triggerAt 在未来。
3. 相对时间与当前可信时间基本一致。
4. 时区合法。
5. 重复规则合法。
6. 离现在异常遥远时要求确认。
7. 低置信度或存在关键 assumptions 时追问。

六、本地离线兜底

模型不可用时仍支持这些确定表达：

- N 秒后
- N 分钟后
- N 小时后
- 半小时后
- 今天 HH:mm
- 明天/后天 HH:mm
- 上午/下午/晚上 N 点
- 每天/每周的明确时间

不要再次出现“一分钟后”连续追问但无法解析的死循环。

多轮规则：

用户：提醒我交报告
AI：请问什么时候提醒？
用户：十分钟后
→ 使用第二条消息的接收时间计算十分钟后
→ 保留第一轮 content="交报告"
→ 直接创建。

九、记忆时间精确到秒

记忆应区分：

- sourceOccurredAt：用户说出信息的时间。
- writtenAt/createdAt：实际写入 SQLite 的时间。
- updatedAt：最后更新的时间。
- timezone：写入时的用户时区。

不要把时间直接拼进 content，使用结构化元数据。

建议统一格式：

2026-07-11T08:09:03Z

并在 UI 显示：

2026-07-11 16:09:03（Asia/Shanghai）

新增数据库 migration，不能修改旧 migration 假装升级：

1. 将 memories.created_at、updated_at 的契约统一为明确 UTC ISO-8601。
2. 增加 source_occurred_at 和 write_timezone 字段。
3. 把旧的 `YYYY-MM-DD HH:mm:ss` UTC 数据安全转换为带 `T` 和 `Z` 的格式。
4. migration 必须事务化、幂等并保留原始秒级时间。
5. 新写入使用 TimeService 提供的时间，不依赖含义模糊的 datetime('now')。
6. 更新记忆时保留 createdAt，只更新 updatedAt。
7. Reflection 队列延迟时：
   - sourceOccurredAt 使用原对话消息时间。
   - writtenAt 使用实际写入时间。
8. 记忆列表 UI 显示写入时间到秒。
9. Prompt 中注入记忆时可显示：
   - [用户陈述时间：...]
   - [记忆写入时间：...]
   但不要让时间标签污染记忆正文。
10. JSON 导出必须包含这些时间字段。
